# 1. Introduction to Terraform State

## What is Terraform State?

Terraform State is a file that stores information about the infrastructure Terraform manages.

Whenever Terraform creates, updates, or deletes resources, it records those resources in a **state file**.

By default, the state file is named:

```text
terraform.tfstate
```

Terraform uses this file to understand:

* Which resources already exist
* Which resources Terraform manages
* Which resources need to be created
* Which resources need modification
* Which resources need deletion

Without the state file, Terraform would have no memory of previously created infrastructure.

---

## Think of Terraform State Like This

Imagine you hire an interior designer to decorate your house.

After decorating, the designer creates a notebook containing:

* Number of rooms
* Furniture locations
* Paint colors
* Electrical appliances

The next time you ask the designer to make changes, they don't inspect the entire house again.

Instead, they read the notebook.

Terraform works exactly the same way.

```
Actual Infrastructure
        ↑
        │
Terraform State
        │
Terraform Configuration
```

The state file acts as the bridge between your Terraform code and the actual cloud infrastructure.

---

# 2. Why Terraform Needs State

Many beginners ask:

> "Terraform already has my `.tf` files. Why does it need another file?"

The answer is simple.

Your `.tf` files describe **what you want**.

The **State File** describes **what currently exists**.

Example:

Configuration:

```hcl
resource "aws_instance" "web" {

  ami           = "ami-123456"

  instance_type = "t2.micro"

}
```

Terraform knows you want an EC2 instance.

But how does it know:

* Instance ID
* Public IP
* Private IP
* ARN
* Volume ID
* Security Group Attachment

Those values are generated by AWS after creation.

Terraform stores them inside the state file.

Without the state file, Terraform would have to recreate resources every time.

---

## Infrastructure Workflow

```
terraform apply
        │
        ▼
AWS Creates Resources
        │
        ▼
Terraform Reads Response
        │
        ▼
terraform.tfstate Updated
```

---

# 3. What is terraform.tfstate?

The `terraform.tfstate` file is a JSON document maintained automatically by Terraform.

You should **never manually edit it** unless specifically instructed by HashiCorp.

Example:

```json
{
  "resources": [
    {
      "type": "aws_instance",
      "name": "web"
    }
  ]
}
```

The actual file is much larger and contains:

* Resource IDs
* Metadata
* Dependencies
* Attributes
* Outputs
* Provider Information

---

## Where is it Located?

Default location:

```
Project Folder

│

├── main.tf

├── variables.tf

├── outputs.tf

├── terraform.tfstate

└── terraform.tfstate.backup
```

---

## Important Files

| File                     | Purpose                      |
| ------------------------ | ---------------------------- |
| terraform.tfstate        | Current Infrastructure State |
| terraform.tfstate.backup | Previous State Backup        |

Terraform automatically creates a backup before updating the current state.

# 4. How Terraform Uses State

Suppose your infrastructure contains:

```
AWS

EC2

S3

VPC
```

Terraform state contains:

```
terraform.tfstate

EC2 ID

S3 Bucket Name

VPC ID
```

Now you modify only the EC2 instance.

Terraform performs the following steps:

```
Read Configuration

↓

Read State File

↓

Compare Differences

↓

Generate Execution Plan

↓

Apply Changes

↓

Update State
```

This process is known as **State Reconciliation**.

---

# 5. Desired State vs Current State

Terraform always compares two things.

## Desired State

Defined inside Terraform code.

Example:

```hcl
instance_type = "t3.micro"
```

---

## Current State

Stored inside:

```
terraform.tfstate
```

---

Terraform performs:

```
Desired State

↓

Compare

↓

Current State

↓

Difference?

↓

Yes

↓

Apply Changes
```

This comparison is the reason Terraform is idempotent.

---

# 6. State File Structure

A simplified state file contains:

```
terraform.tfstate

│

├── Version

├── Terraform Version

├── Serial Number

├── Lineage

├── Outputs

├── Resources

└── Provider Metadata
```

Let's understand each section.

---

## Version

Specifies the format version of the state file.

Example:

```json
"version": 4
```

---

## Terraform Version

Indicates the Terraform version that created the state.

Example:

```json
"terraform_version": "1.12.0"
```

---

## Serial

A counter that increases every time the state changes.

Example:

```
Serial = 10

↓

terraform apply

↓

Serial = 11
```

---

## Lineage

Unique identifier for the state.

Used to ensure Terraform is working with the correct infrastructure.

---

## Outputs

Stores values declared using:

```hcl
output "public_ip" {

 value = aws_instance.web.public_ip

}
```

---

## Resources

The largest section.

Contains:

* Resource IDs
* Resource Type
* Provider
* Attributes
* Dependencies

---

# 7. Why State Makes Terraform Fast

Imagine you manage:

* 500 EC2 Instances
* 100 Load Balancers
* 300 Security Groups
* 200 IAM Roles

Without State:

Terraform would query AWS for every resource before every plan.

That would take several minutes.

With State:

Terraform already knows what exists.

Only changes are evaluated.

Result:

* Faster Plan
* Faster Apply
* Better Performance

---

# 8. Sensitive Data in State

One of the biggest mistakes beginners make is assuming state files are encrypted.

**They are not.**

The state file may contain:

* Passwords
* Database Endpoints
* Access Keys
* Secret Values
* Private IPs
* Public IPs

Example:

```hcl
resource "aws_db_instance" "db" {

 password = "Admin@123"

}
```

Even if the variable is marked as:

```hcl
sensitive = true
```

Terraform hides the value in CLI output.

However, the actual value is still stored inside the state file.

---

## Best Practice

Never:

* Commit state files to GitHub
* Email state files
* Share state files publicly

Instead:

* Use Remote Backend
* Enable Encryption
* Restrict Access

# 9. Terraform State Lifecycle

```
Write Configuration

↓

terraform init

↓

terraform plan

↓

terraform apply

↓

Resource Created

↓

State Updated

↓

Modify Code

↓

terraform apply

↓

State Updated Again
```

The state file evolves throughout the infrastructure lifecycle.

---

# 10. State File Best Practices

## ✔ Store State Remotely

Recommended:

* Amazon S3
* Azure Blob Storage
* Google Cloud Storage
* HCP Terraform

---

## ✔ Enable State Locking

Prevents multiple users from modifying the state simultaneously.

We'll cover this in detail later in this module.

---

## ✔ Never Edit State Manually

Use Terraform commands instead.

Incorrect:

```
Open terraform.tfstate

↓

Delete Resource
```

Correct:

```bash
terraform state rm
```

---

## ✔ Never Store State in Git

Add to `.gitignore`:

```gitignore
*.tfstate

*.tfstate.*

.terraform/
```

---

## ✔ Enable Versioning

If using S3:

Enable Bucket Versioning.

This allows state recovery after accidental deletion.

# 11. Terraform State Commands (Introduction)

Terraform provides several commands to inspect and manage the state.

| Command              | Purpose                                       |
| -------------------- | --------------------------------------------- |
| terraform state list | List all managed resources                    |
| terraform state show | Display detailed information about a resource |
| terraform state mv   | Move a resource within the state              |
| terraform state rm   | Remove a resource from the state              |
| terraform state pull | Download the latest remote state              |
| terraform state push | Upload a local state to the backend           |

The next section of this module will cover each command in detail with examples and hands-on labs.
